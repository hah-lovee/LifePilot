# Life Pilot — инструкция по администрированию

Это руководство для человека, который администрирует развёрнутое приложение: управляет
пользователями, каталогом, инвайт-кодом, инвестиционной интеграцией и обслуживает инфраструктуру
(Postgres, Vault). Архитектурные решения и обоснование стека — в [ARCHITECTURE.md](ARCHITECTURE.md).

## 1. Роль администратора

- Первый зарегистрированный пользователь становится админом автоматически.
- Назначить/снять права у другого пользователя: `/admin` → таблица "Пользователи" → кнопка
  у нужной строки. Доступно только тому, у кого уже есть `is_admin=true`.
- Админ видит дополнительный пункт "Админка" в верхнем меню; обычным пользователям он не показывается.

## 2. Повседневное администрирование

### Инвайт-код
`/admin` → поле "Инвайт-код для регистрации". Хранится в таблице `app_settings` (не в `.env`) —
меняется на лету, без перезапуска backend.

### Каталог привычек, упражнений, групп мышц
`/admin/catalog`:
- Привычки и упражнения, заведённые здесь, видны всем пользователям в отдельном списке "Каталог" —
  это не происходит автоматически, каждый добавляет себе вручную.
- Группы мышц — отдельный управляемый список (не свободный текст), чтобы не плодить дубли вроде
  "Грудь"/"грудь"/"chest".
- У каждого упражнения: "Изменить" (имя + группа мышц), "Заменить фото", "Удалить". Удаление
  каталожной записи не трогает её у пользователей, которые уже добавили себе.
- Фото: только `.jpg/.jpeg/.png/.webp/.gif`, до 3 МБ — проверяется и на фронте, и на backend.

## 3. Модуль "Инвестиции"

### Архитектура
Life Pilot **не хранит** биржевые/брокерские ключи у себя. Они уходят в отдельный сервис
**`trading-keys-api`** (свой git-репозиторий), который держит их в HashiCorp Vault. Backend Life
Pilot — это тонкий прокси: подставляет свой `INVESTMENTS_API_KEY` и `user_guid = str(user.id)`
на каждый запрос, сам ключ/токен пользователя никогда не попадает в базу или логи Life Pilot.

### Первый запуск trading-keys-api
```bash
cd trading-keys-api
docker compose up -d --build
docker logs vault-init   # убедиться, что секреты НЕ напечатаны, видна строка "Root-токен отозван"
```
После первого запуска посмотреть `data/init/tokens.env` — там лежит `API_KEY`. Его нужно
один раз скопировать в `backend/.env` Life Pilot:
```
INVESTMENTS_API_URL=http://localhost:8001   # в проде — внутренний docker-хост, не localhost
INVESTMENTS_API_KEY=<значение API_KEY из tokens.env>
```
Перезапустить backend Life Pilot после изменения `.env`.

### Ежедневный снэпшот капитала
Раз в день (23:50 по времени сервера) встроенный планировщик (APScheduler, живёт в процессе
backend) проходит по всем пользователям с хотя бы одним подключением, забирает баланс и курс
USD/RUB, считает суммарный капитал в рублях и пишет строку в `investment_snapshots` — это и есть
данные для графика "Капитал во времени". Эндпоинты `/summary`, `/diversification`, `/dividends`
сами никогда не пишут в базу — только читают живые данные.

Если backend перезапускался и план поставленных задач пропал — следующий снэпшот будет создан
в 23:50 автоматически, ничего вручную делать не нужно.

### Ротация ключа доступа AppRole (раз в ~90 дней)
У `trading-keys-api` секрет (`secret_id`), которым backend логинится в Vault, живёт 90 дней.
Когда подходит срок:
```bash
cd trading-keys-api
docker compose run --rm -v "$(pwd)/data/init:/data/init" python:3.11-slim \
  sh -c "pip install requests -q && python /scripts/rotate_secret_id.py"
docker compose restart api
```
Это не требует root-токена (он отозван навсегда после первой инициализации) — используется отдельный
узкоправный `VAULT_ROTATOR_TOKEN`, который умеет только перевыпускать `secret_id`.

### Если нужно отключить пользователю биржу/брокера руками
Через UI: `/investments/connections` → кнопка с крестиком у подключения. Если недоступно через UI —
прямой вызов API с правами админа не нужен, обычный пользователь и так управляет только своими
подключениями (`user_guid` подставляется backend'ом из текущей сессии, подменить нельзя).

## 4. Бэкапы

| Что | Где | Чем бэкапить |
|---|---|---|
| Данные приложения (дневник, привычки, спорт, снэпшоты капитала) | Postgres `life_pilot` | `pg_dump`/снэпшот volume `postgres_data` |
| Биржевые/брокерские ключи | Vault (`trading-keys-api/data/vault`) | бэкап файлового хранилища Vault — **обязательно вместе с** `data/init/tokens.env` (без unseal-ключей бэкап Vault бесполезен) |
| Фото упражнений | `backend/uploads/` (volume в проде) | обычный файловый бэкап |

Восстановление Vault без `tokens.env` невозможно — там лежат unseal-ключи (2 из 3 нужно ввести
вручную). Храните этот файл отдельно от обычных бэкапов БД, с ограниченным доступом.

## 5. Безопасность — что уже сделано и на что обращать внимание

- Vault: unseal на 3 ключа с порогом 2 (не один ключ), root-токен отзывается сразу после
  настройки, `secret_id` у AppRole не бессрочный (90 дней, ротация — см. выше), секреты не
  печатаются в `docker logs`.
- Каждый инвестиционный эндпоинт Life Pilot сам подставляет `user_guid` из аутентифицированной
  сессии — пользователь не может передать чужой id и увидеть чужой портфель.
- Пароли — bcrypt, минимум 8 символов при регистрации. JWT — 7 дней жизни (рассчитано на
  личное/семейное использование, не публичный сервис).
- Загрузка фото — белый список расширений + лимит 3 МБ проверяются на backend (не только на фронте).
- `trading-keys-api` и Vault **не должны** иметь публичный порт в проде — только внутренняя
  docker-сеть, к которой обращается backend Life Pilot. Сейчас в `infra/docker-compose.yml`
  Life Pilot эти сервисы не прописаны — при продакшен-деплое нужно поднять `trading-keys-api`
  как отдельный compose-стек на той же docker-сети и указать `INVESTMENTS_API_URL` на его
  внутреннее имя контейнера (не `localhost`).
- Регистрация закрыта инвайт-кодом — не публичная.
- Нет рейт-лимита на `/api/auth/login` — для закрытого личного/семейного сервиса риск
  принят осознанно; если планируется более широкий доступ, стоит добавить (например, `slowapi`).

## 6. Локальная разработка — порты

| Сервис | Порт |
|---|---|
| Life Pilot backend | 8002 |
| Life Pilot frontend | 3000 |
| trading-keys-api | 8001 |
| Vault UI/API (trading-keys-api) | 8200 |
| Postgres (Life Pilot) | 55432 (или как настроено локально) |

Порт 8000 сознательно не используется backend'ом Life Pilot — см. раздел 7.

## 7. Известные проблемы при локальной разработке (Windows)

**`uvicorn --reload` оставляет процессы-зомби.** На Windows воркер, который `--reload`
перезапускает при изменении файла, иногда не завершается при попытке остановить родительский
процесс — он живёт дальше под отдельным PID (видно в `netstat -ano`, но команда `taskkill`
иногда не находит его по обычным фильтрам, если искать по пути к проекту в командной строке:
у воркера-зомби в командной строке пути может не быть вообще). Внешне это выглядит как
"настройки не применяются после перезапуска" или "порт всё ещё занят". Если backend ведёт
себя так, будто не подхватил свежий код:
```powershell
netstat -ano | findstr ":8002"          # посмотреть все PID на порту
taskkill /F /PID <pid>                  # убить каждый найденный PID по очереди
```
Для долгоживущего dev-процесса предпочтительнее запускать без `--reload` и перезапускать вручную
после правок — так зомби не накапливаются.
